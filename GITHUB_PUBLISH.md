# Как опубликовать проект BuildIntel на GitHub

## Пошаговая инструкция

### Шаг 1: Создайте репозиторий на GitHub

1. Перейдите на https://github.com
2. Войдите в свой аккаунт (или создайте новый)
3. Нажмите кнопку **"+"** в правом верхнем углу → **"New repository"**
4. Заполните форму:
   - **Repository name**: `buildintel` (или другое название)
   - **Description**: "AI ассистент для анализа маркетинга в строительстве"
   - Выберите **Public** (публичный) или **Private** (приватный)
   - **НЕ** отмечайте галочки "Add a README file", "Add .gitignore", "Choose a license" (мы уже создали эти файлы)
5. Нажмите **"Create repository"**

### Шаг 2: Инициализируйте Git в проекте

**Перед началом:** Убедитесь, что файл `.env` существует и содержит ваш реальный API ключ. Этот файл **НЕ** будет опубликован благодаря `.gitignore`.

Откройте терминал в корневой директории проекта (`monitor_kp`):

```bash
# Инициализация Git репозитория
git init

# Проверьте, что .env не будет добавлен (должен быть в .gitignore)
git status

# Добавление всех файлов (кроме тех, что в .gitignore)
git add .

# Первый коммит
git commit -m "Initial commit: BuildIntel - AI ассистент для строительства"
```

### Шаг 3: Подключите локальный репозиторий к GitHub

GitHub покажет вам команды после создания репозитория. Выполните:

```bash
# Добавьте удаленный репозиторий (замените YOUR_USERNAME на ваш GitHub username)
git remote add origin https://github.com/YOUR_USERNAME/buildintel.git

# Или если используете SSH:
# git remote add origin git@github.com:YOUR_USERNAME/buildintel.git

# Отправьте код на GitHub
git branch -M main
git push -u origin main
```

### Шаг 4: Проверьте результат

1. Обновите страницу репозитория на GitHub
2. Вы должны увидеть все файлы проекта
3. README.md автоматически отобразится на главной странице репозитория

## Что будет опубликовано

Благодаря файлу `.gitignore`, следующие файлы **НЕ** будут опубликованы:

- `.env` - файл с API ключами (важно для безопасности!)
- `venv/` - виртуальное окружение
- `__pycache__/` - кэш Python
- `*.pyc` - скомпилированные файлы Python
- `dist/` и `build/` - папки сборки .exe
- `history.json` - история запросов (личные данные)
- `chromedriver-win64/` - ChromeDriver (большие бинарные файлы)
- `.env.example` - будет опубликован как шаблон (без реальных ключей)

## Важные моменты безопасности

### ⚠️ ВАЖНО: Не публикуйте API ключи!

Перед публикацией убедитесь, что файл `.env` находится в `.gitignore`:

```bash
# Проверьте содержимое .gitignore
cat .gitignore

# Должна быть строка:
.env
```

Если файл `.env` случайно был добавлен в Git:

```bash
# Удалите его из индекса Git (но оставьте локально)
git rm --cached .env

# Сделайте коммит
git commit -m "Remove .env from repository"
```

### Файл .env.example уже создан

Файл `.env.example` уже создан в проекте и будет опубликован как шаблон для других пользователей (без реальных API ключей).

## Обновление проекта на GitHub

После внесения изменений в проект:

```bash
# Проверьте статус изменений
git status

# Добавьте измененные файлы
git add .

# Сделайте коммит с описанием изменений
git commit -m "Описание ваших изменений"

# Отправьте изменения на GitHub
git push
```

## Полезные команды Git

```bash
# Просмотр истории коммитов
git log

# Просмотр изменений
git diff

# Просмотр статуса
git status

# Отмена последнего коммита (но сохранение изменений)
git reset --soft HEAD~1

# Просмотр удаленных репозиториев
git remote -v
```

## Альтернатива: Использование GitHub Desktop

Если не хотите использовать командную строку:

1. Скачайте GitHub Desktop: https://desktop.github.com/
2. Установите и войдите в свой аккаунт
3. Нажмите **"File"** → **"Add Local Repository"**
4. Выберите папку `monitor_kp`
5. Нажмите **"Publish repository"**
6. Выберите название и видимость (Public/Private)
7. Нажмите **"Publish Repository"**

## Что делать после публикации

1. **Добавьте описание проекта** на странице репозитория
2. **Добавьте теги (tags)** для версий: `v1.0.0`
3. **Создайте Issues** для отслеживания багов и предложений
4. **Добавьте лицензию** (если нужно) - создайте файл `LICENSE`
5. **Настройте GitHub Pages** (опционально) для документации
